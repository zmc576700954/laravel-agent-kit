# Project Memory

用于在本仓库内协作时快速校准上下文，避免把当前仓库误判为业务 Laravel 应用。

## 1. 仓库身份

- 本仓库是 `Laravel Agent Kit`：一个可分发的 Laravel Agent 规则包仓库。
- 目标是维护一套 vendor-neutral 源文件，再产出适配不同 AI 工具的安装产物。
- 当前仓库重点是“规则、技能、适配、分发”，不是业务功能、数据库表或应用接口实现。

## 2. 目录边界

- `global/`：跨项目通用规则，是 Source of Truth 的一部分。
- `project/`：当前仓库绑定的操作性规则与附属资产。
- `skills/`：场景 Skill / Profile Skill 的 Source of Truth。
- `profiles/`：legacy Profile 文档，默认以 `skills/` 中对应内容为准。
- `adapters/`：各 AI 工具的入口映射、安装方式、限制与兼容性说明。
- `dist/`：面向用户的可安装产物，只作为分发结果，不作为编辑入口。
- `docs/`：安装、结构、维护、发布、示例等说明文档。

## 3. 真相源与非真相源

- 优先阅读并修改：`global/`、`project/`、`skills/`、`profiles/`（legacy）。
- 优先参考：`docs/source-of-truth.md`、`docs/structure.md`、`README.md`。
- 不把 `dist/` 当作规则编辑入口；它是由源文件拼装/映射后的产物。
- 工具专用安装路径、入口文件名、兼容性差异，优先放在 `adapters/` 或 `docs/`，不要反写进源规则。

## 4. 协作工作方式

- 先调查，再修改：至少检索一次并读取关键文件。
- 默认最小改动，不顺手扩改、不顺手重构。
- 优先复用已有模式，不重复造轮子。
- 中/重任务先列 To-do，再进入实现；步骤至少覆盖 Read / Change / Verify。
- 输出应围绕当前仓库的规则体系、技能结构、安装分发与维护约束展开。

## 5. 当前仓库内的关键约束

- 若任务涉及工作流概念，优先显式给出 `Scene`、`Stage`、`TaskId`，不要强推断。
- Profile 默认不启用；只有满足条件且被显式声明时才生效。
- `skills/` 只写场景差异与闭环步骤，不重复大段 Global / Project 通用规则。
- 源文件保持 vendor-neutral，避免绑定具体工具名称、安装路径或工具专用导入语法。

## 6. 避免跑偏

- 不把本仓库当成业务系统去寻找控制器、模型、迁移、接口实现。
- 不默认修改 `dist/` 来修“规则问题”；应回到源文件目录修正。
- 不假设存在未读到的业务代码；如果任务需要额外上下文，先明确说明缺失点。
