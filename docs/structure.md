# 仓库结构与模块标准

## 顶层目录职责边界

- `global/`：通用规则（baseline + reality layer + 模板 + checklist）。不得包含任何具体项目名词、工具入口文件名、安装路径。
- `project/`：仓库绑定的操作性规则与附属资产（脚本、模板、任务记录规范等）。允许出现仓库目录约定与执行入口脚本，但不应包含 `dist/**` 的工具产物引用。
- `profiles/`：可选规则包（Profile）。允许包含项目/场景特有不变量，但必须明确启用条件，且默认不启用。
- `adapters/`：面向工具的“入口映射层”。定义每个工具的入口文件、安装路径、组合方式、限制与兼容性。
- `dist/`：面向用户的“可安装产物”。应做到复制/粘贴即可用；不作为编辑入口。
- `docs/`：安装、升级/回滚、示例、维护规范与产物生成说明。

## 术语与优先级（统一约定）

- Global Rules：跨项目稳定治理与 Laravel Baseline 底线（`global/`）
- Project Rules：仓库绑定的操作性规范与资产（`project/`）
- Skills：按场景闭环剧本（`skills/`）
- Profile：特定栈/特定协作协议的增强包（`skills/*` 的 profile kind；legacy 在 `profiles/`）

优先级（高 → 低）：

1. Hard（Global Reality Layer 的安全/可控性约束）
2. Project Rules（仓库内工作方式约束）
3. Profile（默认不启用，显式启用后生效）
4. Baseline（Laravel 技术底线与工程惯例）

## 常见边界问题（避免重复与漂移）

- 不要把“安装路径/入口文件名/工具专用目录结构”写进 `global/`、`project/`、`skills/`
- `skills/` 不重复粘贴 Global/Project 的大段通用规则，只写场景差异与闭环步骤
- `dist/` 只作为产物，避免在源文件中引用 `dist/**`

## Profiles 最小模块标准

每个 Profile 文件必须包含：

1. **Profile 标题与用途**：一句话说明解决的协作/治理问题
2. **何时启用**：显式启用方式与触发条件（默认不启用）
3. **Hard / Soft（可选）**：项目不变量与可覆盖偏好
4. **最小验证口径**：至少 1 条可执行命令或最小人工验证清单

禁止在 Profile 中复制 `global/` 内容；只写差异与额外约束。

## Adapters 最小模块标准

每个 Adapter 文档必须包含：

1. **目标工具**：工具名与入口识别方式（入口文件名、目录结构）
2. **安装路径**：项目级与全局级（如工具支持）分别如何落地
3. **组合规则**：global 如何生效、profile 如何启用、优先级与冲突处理
4. **限制与兼容性**：工具不支持的能力（如 include/import、skills 安装方式差异）
5. **升级/回滚注意点**：如何不覆盖项目自定义、如何对比变更
